## Context

The app today delivers messages live via Centrifugo WebSocket. When the tab is unfocused or closed, users miss activity entirely — there is no OS-level surface. Modern browsers expose two complementary standards:

- **Notifications API** (`window.Notification`) — page-originated, works only while a tab with live JS is open. Ideal for "you have an unfocused tab with the chat".
- **Web Push API** + **Service Worker** — server-originated, delivered by the browser's push service (FCM for Chromium, Mozilla autopush for Firefox, APNs for Safari on iOS 16.4+ *installed as PWA*), survives the tab being closed.

Because iOS only honors Web Push for sites installed as a PWA (manifest + `display: standalone` + user-initiated add-to-home-screen), and because installability also improves desktop ergonomics, this change bundles a minimal PWA shell with the notification work rather than splitting it. The `web-push` npm library is the de-facto standard for VAPID-authenticated push from Node.

## Goals / Non-Goals

**Goals:**

- Surface DMs, @-mentions, and friend requests as OS notifications within the 3 s delivery budget whether the tab is focused, backgrounded, or closed.
- Give users granular, persistent control (categories + mute window) without a server round-trip on every message.
- Make the app installable as a PWA on desktop (Chrome/Edge) and iOS 16.4+.
- Keep Centrifugo as the authoritative live transport; push is a **strictly additive** fallback.
- Zero impact on users who never enable notifications — no permission prompts on load, no service-worker-driven caching of message history.

**Non-Goals:**

- Offline chat UX (reading/sending messages while disconnected). The service worker's only offline surface is a minimal fallback page; history is not cached.
- Native apps (Electron, Capacitor, Tauri).
- Push providers other than the browser's own push service — no OneSignal, no Firebase Cloud Messaging SDK. VAPID + `web-push` only.
- In-page toast/sound notifications (out of scope; may be a separate follow-up).
- Notification grouping / rich actions beyond "click to focus". No inline reply from the notification for R4-era scope.

## Decisions

### D1. Two-path delivery: Notification API (foreground) + Web Push (background)

**Chosen:** Dispatch via Notification API from the Centrifugo subscription handler when the tab is alive, and via Web Push from the server when the recipient has no live Centrifugo connection *or* every connection is in a hidden/unfocused state.

**Alternatives considered:**

- *Push for everything*: simpler, but wastes the browser's push-service quota, shows duplicate notifications when the tab is already open, and adds 100–500 ms latency compared to the in-page path.
- *Notification API only*: fails the moment the tab is closed, which is the single most requested use case.

**Signal used to decide "is the user present?":** Centrifugo presence for the user's channel plus a per-connection `focused` flag the client publishes to a server-side presence key. If any connection is `focused`, no push is sent. If all are `hidden` or `0 connections`, push is sent. Rationale: avoids double notifications and matches the "AFK = all tabs idle > 1 min" rule already in the app.

### D2. Preferences live client-side (IndexedDB via `idb-keyval`), mirrored to server only for push categories

**Chosen:** Category toggles and mute windows are stored in IndexedDB and queried in the Centrifugo handler without server round-trip. The server keeps a lightweight mirror (only the categories relevant to push dispatch: DM on/off, mentions on/off, per-room on/off, friend-requests on/off, global mute-until) on the `PushSubscription` row so the server can decide whether to send a push without querying the client.

**Why not fully server-side:** Foreground dispatch is hot path; a DB round-trip per incoming message at 300 CCU is avoidable.

**Why not fully client-side:** The server must know category/mute state for background push, because by definition the client isn't reachable.

### D3. Service worker authored as static `public/sw.js`, not generated by next-pwa

**Chosen:** Hand-written `public/sw.js` registered from the root layout. It handles `push`, `notificationclick`, and a minimal `fetch` fallback.

**Why not next-pwa / @serwist/next:** Those tools aggressively precache routes and assets, which conflicts with "history stays on the server" and with Next 15's streaming/RSC model. We want a scoped-down SW that does *only* push + click; a 60-line file is clearer than a plugin we have to configure out of caching everything.

### D4. VAPID keys baked at build for public half, env-only for private half

- `NEXT_PUBLIC_VAPID_PUBLIC_KEY` is exposed to the browser (safe; it's the public half by design).
- `VAPID_PRIVATE_KEY` is server-only.
- `VAPID_SUBJECT` is a `mailto:` URI required by the RFC; default to the ops contact.
- Both compose files take these as env vars; `docker-compose.prod.yml` treats the private key as a required secret via `${VAR:?}`.

### D5. Click-to-focus contract

Every notification payload carries `{ type: 'dm' | 'mention' | 'room' | 'friend-request', url: '/chat/...' | '/contacts' | ... , tag: string }`. The SW's `notificationclick` handler:

1. Looks for an existing client window (`clients.matchAll({ type: 'window', includeUncontrolled: true })`).
2. If one matches the target origin, `focus()` it and postMessage `{ navigate: url }` — the app's client router handles in-app navigation without a full reload.
3. Otherwise `clients.openWindow(url)`.

`tag` is used to coalesce repeated notifications for the same conversation so a burst of 10 messages collapses to one row.

### D6. Notification sounds: page-level `HTMLAudioElement` with gesture-unlock

**Chosen:** Preload one short OGG (with WAV fallback) per category into an `<audio>` element pool at app start. On the same user gesture that grants notification permission, call `audio.play(); audio.pause(); audio.currentTime = 0` for each element to satisfy the autoplay policy ("unlock" step). Subsequent event-driven plays then succeed in Chrome, Firefox, and Safari (desktop + iOS PWA).

**Why `HTMLAudioElement` over Web Audio API:** dramatically simpler (no AudioContext lifecycle, no decode step on first use), small clips preload fine, and volume-per-element is sufficient. Web Audio would only matter if we needed mixing, effects, or sample-accurate timing — none of which apply to a "ping" on message arrival.

**Autoplay policy mitigations:**

- **Gesture-unlock on permission grant**: the most reliable trigger; the same click that grants `Notification` permission also primes the audio pool.
- **Fallback gesture-unlock on first document click**: a one-shot capture-phase `click` listener unlocks the pool on the first interaction anywhere, in case the user enables notifications in browser settings rather than via our button (so the permission-grant gesture never fires in our code).
- **`Navigator.getAutoplayPolicy()` probe** (when available) reports `allowed` / `allowed-muted` / `disallowed`; when `disallowed`, we skip playback silently rather than log an error for every message.
- **Per-category throttle**: at most one sound per category per **1500 ms** (per 500 ms for mentions). Prevents audible stuttering during burst fan-out.
- **No sound when `document.visibilityState === 'visible'` AND the conversation is open**: the user is already looking at the message; a ping would be noise.

**Why no sound for background (push-delivered) notifications:** the service worker cannot instantiate `HTMLAudioElement` or `AudioContext`. The OS-level notification surface (which the SW triggers via `registration.showNotification`) already plays the operating system's own notification sound respecting user OS-level mute, DND, focus modes, etc. — that is the correct behavior and we should not try to double it.

**Assets:** ship short (≤ 1 s), normalized, royalty-free clips committed to `public/sounds/` as OGG Vorbis (universal in modern browsers). No WAV fallback needed in target browsers (Chrome/Edge/Firefox/Safari 14+ all decode OGG). Total asset budget ~40 KB.

### D7. Subscription lifecycle

- On permission grant, the client calls `registration.pushManager.subscribe({ userVisibleOnly: true, applicationServerKey })` and POSTs the resulting subscription to `/api/push/subscribe`.
- The server upserts keyed by `(userId, endpoint)`.
- On every push attempt, `web-push` failures with status `404` or `410` cause the row to be deleted.
- On sign-out, the client calls `DELETE /api/push/subscribe` with the current endpoint *and* `pushSubscription.unsubscribe()`. Other sessions' subscriptions are untouched (per the sign-out rule).

## Risks / Trade-offs

- **Safari/iOS quirks** → Mitigation: gate UI with feature detection; show "Install this app to enable notifications" hint on iOS Safari when not installed.
- **Browser quietly denies after one dismiss** → Mitigation: never prompt on load; only on explicit button click; if `permission === 'denied'` show a help link to browser settings instead of re-prompting.
- **Autoplay policy silently blocks the first ping** → Mitigation: gesture-unlock on permission grant + fallback one-shot `click` unlock listener + `getAutoplayPolicy()` probe to detect `disallowed` and surface a one-time tooltip ("Click anywhere to enable sound") instead of pretending it works.
- **Duplicate notifications (foreground + push race)** → Mitigation: D1's presence/focused-flag gate; additionally, every push payload carries `tag` so the OS collapses any race-window duplicates.
- **Push service outages (FCM/Mozilla/APNs)** → Mitigation: push is explicitly best-effort; Centrifugo remains authoritative, and unread badges from the existing `unread-badges` capability cover the "missed it" case on next visit.
- **Service worker cache invalidation** → Mitigation: SW doesn't cache app code or history; only the offline fallback page. Bump `CACHE_VERSION` constant on SW changes; SW uses `self.skipWaiting()` + `clients.claim()` so new versions take effect on next navigation.
- **VAPID private key leakage** → Mitigation: server-only env, never logged; rotate via redeploy (all existing subscriptions become invalid and are pruned on first 410).
- **iOS requires PWA install** → Mitigation: documented in README; settings panel shows an iOS-specific "Add to Home Screen" instruction when installability criteria aren't met.

## Migration Plan

1. Ship Prisma migration adding `PushSubscription` (empty table is safe; nothing depends on it yet).
2. Deploy with VAPID vars set but feature-flagged off in the UI (`NEXT_PUBLIC_NOTIFICATIONS_ENABLED=false`).
3. Smoke-test in production via staff accounts; flip the flag on.
4. Rollback: flip the flag off; no data migration needed. Existing subscriptions stay in the DB and resume working when re-enabled.
